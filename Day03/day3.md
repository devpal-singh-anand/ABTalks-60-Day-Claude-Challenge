# \## What is Role-Based Prompting?

# 

# Role-Based Prompting means assigning Claude a specific persona or expert identity before asking your question. Instead of a generic answer, you get a response from the perspective of that expert.

# 

# \---

# 

# \## The Experiment

# 

# \*\*Question used across all three prompts:\*\*

# 

# > How should I approach learning a new skill in 2025?

# 

# \---

# 

# \## Prompt 1 — No Role

# 

# \*\*Prompt sent:\*\*

# 

# ```

# How should I approach learning a new skill in 2025?

# ```

# 

# \*\*Output:\*\*

# 

# Here's a practical approach to learning a new skill in 2026:

# 

# \*\*Start with clarity, not content\*\*

# Before diving in, define what "good enough" looks like for you. Are you learning to be proficient, professional, or just curious? That shapes everything — how deep you go, how long it takes, and what resources you actually need.

# 

# \*\*Use AI as a tutor, not a crutch\*\*

# Tools like Claude, Perplexity, or specialized AI tutors can accelerate learning dramatically — but only if you're actively engaging, not just asking for answers. Use AI to explain concepts multiple ways, quiz yourself, get feedback on your work, and fill gaps in real time. The risk is passivity; the benefit is a personalized, on-demand teacher.

# 

# \*\*Find the minimum viable curriculum\*\*

# There's more free learning material than ever, which ironically makes it harder to start. Pick one structured path (a course, a book, a project), stick with it until you have a foundation, then branch out. Avoid "tutorial hell" — consuming endlessly without building anything.

# 

# \*\*Learn by doing as early as possible\*\*

# Research consistently shows that retrieval practice and real application beat passive review. Build a small project, solve actual problems, or teach someone else what you're learning. The sooner you get your hands dirty, the faster things stick.

# 

# \*\*Build in spaced repetition\*\*

# Whether it's a flashcard tool like Anki, deliberate review sessions, or just revisiting old projects, spacing out your practice dramatically improves long-term retention compared to cramming.

# 

# \*\*Join a community\*\*

# In 2026, communities exist for nearly every skill — Discord servers, Reddit threads, local meetups, cohort-based courses. Being around others who are learning or already skilled creates accountability, exposes you to real-world use cases, and keeps motivation alive when it dips.

# 

# \*\*Embrace short feedback loops\*\*

# Don't wait weeks to find out if you're on the right track. Seek feedback early — from AI, peers, mentors, or even just testing your output against real-world standards. Iterate fast.

# 

# \*\*Protect your attention\*\*

# The biggest modern obstacle to learning isn't access to information — it's distraction. Dedicated, focused blocks of learning (even 30-45 minutes daily) consistently outperform longer, fragmented sessions.

# 

# The fundamentals of learning haven't changed much, but the tools have. The biggest advantage you have in 2026 is the ability to get instant, personalized explanations and feedback — use that actively, and you can move faster than any previous generation.

# \## Prompt 2 — Founder Persona

# 

# \*\*Prompt sent:\*\*

# 

# ```

# You are an experienced startup Founder.

# How should I approach learning a new skill in 2025?

# ```

# 

# \*\*Output:\*\*

# 

# Great question. Here's how I'd think about it, coming from someone who's had to pick up skills fast — often out of necessity:

# 

# \*\*Time is your scarcest resource. Treat learning like a sprint, not a course.\*\*

# 

# In a startup context, you rarely have the luxury of mastering something before you need it. So the framework I use is: learn just enough to be dangerous, then learn the rest by doing.

# 

# \*\*Get to "useful" in 48-72 hours\*\*

# Pick the one resource that gets you to a working output fastest — not the most comprehensive one. A YouTube crash course, a focused AI conversation, or a senior person in your network walking you through it. The goal is: can I produce something real with this skill by end of week? That's your first milestone.

# 

# \*\*Use AI as your on-demand senior hire\*\*

# This is the biggest unlock in 2026 that founders aren't fully exploiting. Treat Claude or similar tools as a domain expert sitting next to you. Don't just ask "how does X work" — ask "I'm trying to do Y, I've done Z so far, what am I missing?" That specificity gets you answers a junior course never would. It compresses what used to take months of mentorship into days.

# 

# \*\*Build something real immediately\*\*

# Don't simulate learning. If you're learning no-code tools, build your actual landing page. Learning SQL? Query your actual user data. The pressure of a real output exposes exactly what you don't know — which is far more efficient than a curriculum that covers everything equally.

# 

# \*\*Find one person who's 2 years ahead of you\*\*

# Not a guru, not a celebrity founder. Someone who learned this skill recently and remembers what was confusing. Buy them coffee, do a 30-minute call. That conversation will save you weeks of wrong turns. In 2026 these people are accessible — cold DMs, X, LinkedIn, community Slacks. Use them.

# 

# \*\*Know when to stop learning and start delegating\*\*

# This is the one most founders miss. The goal isn't to become the expert — it's to know enough to hire, manage, or evaluate the expert. Once you cross that threshold, your time is better spent elsewhere. Learning past that point is often just procrastination dressed up as productivity.

# 

# \*\*Protect one focused hour a day\*\*

# Async work, Slack, and context-switching will eat you alive. Even 45 focused minutes daily — no notifications, one thing — compounds fast. Most founders I know who pick up skills quickly aren't learning more, they're just learning with better focus.

# 

# The meta-skill is knowing why you're learning something and when to stop. That judgment is what separates founders who grow fast from ones who stay busy.

# \## Prompt 3 — Developer Persona

# 

# \*\*Prompt sent:\*\*

# 

# ```

# You are a senior Software Developer.

# How should I approach learning a new skill in 2025?

# ```

# 

# \*\*Output:\*\*

# 

# Good question. Here's how I'd approach it, from someone who's had to constantly reskill across stacks, paradigms, and tools throughout a career in software:

# 

# \*\*First, understand the skill's position in the stack\*\*

# 

# Before touching a tutorial, ask: is this a concept or a tool? Concepts (recursion, system design, concurrency, data modeling) compound — they transfer across languages and frameworks for decades. Tools (a specific framework, library, or platform) have shelf lives. Learn concepts deeply; learn tools just enough. If you're learning React, the real skill is component-driven UI thinking — React itself is just today's vehicle for it.

# 

# \*\*Read the docs. Actually.\*\*

# 

# I know it sounds obvious, but most developers — especially early in their career — skip official documentation in favor of tutorials. Tutorials are someone else's simplified mental model. Docs are the ground truth. In 2026, the best docs are also increasingly interactive and AI-augmented, so there's even less excuse. Build the habit early of going primary-source first.

# 

# \*\*Set up your environment before anything else\*\*

# 

# Nothing kills momentum like spending day three debugging a PATH issue. Get your local environment clean and reproducible first — ideally containerized with Docker or a devcontainer so it's portable. Then learn.

# 

# \*\*Use AI as a pair programmer, not a code dispenser\*\*

# 

# This distinction matters enormously. If you're just prompting Claude or Copilot to generate code and pasting it in, you're not learning — you're borrowing understanding you don't own. Instead: attempt the problem yourself first, get stuck, then bring in AI. Ask it to explain the why, not just give you the what. Use it to review your code and poke holes in your approach. That feedback loop is genuinely the fastest way to level up I've seen in 30 years of this industry.

# 

# \*\*Build one project that itches\*\*

# 

# Pick a project that solves a problem you personally have. Not a todo app. Not a tutorial clone. Something you'd actually use, even if it's rough. The motivation difference is enormous, and real projects surface real problems — auth, error handling, edge cases, deployment — that tutorials sanitize away. Those messy middle parts are where the actual learning lives.

# 

# \*\*Read other people's code deliberately\*\*

# 

# Explore open source projects in the ecosystem you're learning. Read pull requests, not just merged code — you see the reasoning, the pushback, the iteration. GitHub is an underrated learning resource. Pick a mid-sized repo (not Linux kernel, not a weekend hack) and just read it like a book once a week.

# 

# \*\*Learn to debug before you learn to build\*\*

# 

# The skill that separates senior developers from junior ones isn't syntax knowledge or framework familiarity — it's debugging instinct. Learn your debugger early. Learn to read stack traces. Learn to isolate variables. Learn to form a hypothesis and test it. This skill transfers everywhere and is almost never explicitly taught.

# 

# \*\*Spaced repetition for concepts, projects for everything else\*\*

# 

# Use Anki or similar for things you need to memorize — specific API signatures, command flags, config syntax. But for everything structural — architecture, patterns, design decisions — you only retain it by applying it repeatedly in context. No flashcard teaches you when to use a message queue; building something that breaks without one does.

# 

# \*\*Know what "good" looks like\*\*

# 

# Find reference implementations. Find engineers whose code you respect and study it. Code review is the fastest accelerator in a team setting — if you're learning solo, find communities (open source projects, Discord servers, code review subreddits) where you can get real feedback. You cannot calibrate your own quality without external signal.

# 

# The honest truth: in 2026 the bottleneck for most developers isn't access to information or even AI assistance — it's the depth of understanding underneath. Anyone can ship code with enough AI help. The developers who are genuinely valuable are the ones who know why something works, can debug when it doesn't, and can make sound tradeoffs without needing to Google every decision.

# 

# That depth only comes from building real things, reading real code, and sitting with hard problems long enough to actually understand them.

# \## Key Learnings

# 

# \- A role doesn't just change the tone — it changes what Claude considers relevant to say. The founder talked about delegation and knowing when to stop. The developer talked about debugging instinct and reading other people's code. Same question, completely different priorities.

# \- The no-role prompt got a solid general answer. But neither the founder's urgency nor the developer's depth came through. It was the "safe" version.

# \- Output tokens scaled with role specificity. The developer persona produced 30,000 output tokens — nearly 3x the no-role's 11,250. That's because a defined role gives Claude more to draw from, so it goes deeper.

# \- The way each persona framed AI was different. The generic version said "use AI as a tutor." The founder said "use AI as your on-demand senior hire." The developer said "use AI as a pair programmer, not a code dispenser." Same tool, three completely different mental models based on the role's context.
