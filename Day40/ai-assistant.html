Documentation

Why Brief exists

Most people who deal with contracts, vendor agreements, policy documents, or dense reports aren't lawyers or analysts — they need a fast, honest read on risk and obligations, not a wall of caveats. Brief is built around one workflow: paste or upload a document, get a structured, decision-ready breakdown back.

From blueprint to product

The product discovery phase identified document review as a recurring, high-stakes, time-pressured task with a clear "before/after" — a person arrives confused about a document and leaves with a decision. That shaped the scorecard/checklist output style and the risk-first personality, rather than a general-purpose chatbot.

How the system prompt works

Every request to the model is prefixed with a hidden system prompt that defines Brief's identity, responsibilities, out-of-scope boundaries, reasoning style, and output structure. You can inspect the exact text in Developer Mode.

Conversation architecture

Each session keeps its own message history in your browser's local storage. When memory is enabled, prior turns in the active session are sent along with each new request so the assistant keeps context; disable it to send each message in isolation.

API integration

Brief calls your chosen provider directly from your browser using your own API key, stored only in local storage on this device. Nothing passes through a Brief-owned server. Supported providers: Anthropic (Messages API), OpenAI (Chat Completions), Groq (OpenAI-compatible, ultra-fast open models), and OpenRouter (routes to almost any model with one key).

Document parsing (non-AI)

PDF and DOCX text is extracted entirely on your device before anything reaches the model — pdf.js walks each page's text layer, and Mammoth reads the DOCX XML — so no AI call is spent on the mechanical work of getting text out of a file. The extracted text is inserted into your message as plain text; the model never sees the original binary. Scanned/image-only PDFs have no text layer and will report zero characters extracted.

Findings panel & dashboard

When an analysis produces risk items, obligations, or recommendations, the model appends a small structured JSON block after its written answer. Brief parses that block, hides it from the conversation, and uses it to render citation-style Findings cards under the reply and to populate the Dashboard tab — a running Risk Score, Obligations & deadlines, Missing clauses, and Recommendations aggregated across the whole session.

UI decisions

The seal-and-margin motif nods to document annotation — a wax-seal mark for identity, a gold margin rule on assistant replies standing in for a reviewer's pen. Everything else stays quiet so the content, not the chrome, carries attention.

Where this goes next

  • Retrieval (RAG) — index long or multi-document uploads so answers cite the exact source passage.
  • Multi-agent review — a drafting pass and a separate adversarial "red team" pass that argues the counterparty's position.
  • Voice — dictate questions and hear scorecards read back for accessibility and hands-free review.
  • Persistent cloud memory — sync sessions across devices instead of per-browser local storage.
